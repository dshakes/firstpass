# Runbook: cutting a release

Firstpass releases are built by [`cargo-dist`](https://opensource.axo.dev/cargo-dist/),
configured in [`dist-workspace.toml`](../../dist-workspace.toml) and driven by
[`.github/workflows/release.yml`](../../.github/workflows/release.yml)
(autogenerated by `dist`, do not hand-edit). **As of this writing that
workflow has never been run** â€” there is no published release yet. This
runbook is the procedure for when one is cut, not a report that one has been.

The tag push in step 4 is the trigger for a real GitHub release with
downloadable artifacts. **Treat it as a human-gated action** â€” nothing in CI
pushes a version tag on its own, and you should not automate that step
without deciding to.

## Prerequisites

- `dist` installed locally and matching the pinned version in
  `dist-workspace.toml` (`cargo-dist-version = "0.32.0"`):
  ```bash
  cargo install cargo-dist@0.32.0 --locked
  dist --version   # should print: cargo-dist 0.32.0
  ```
- Push access to `main` and permission to push tags (this is the
  human-gated step â€” see above).
- A clean `main` at the commit you intend to release: `git status` shows
  nothing pending, and CI (`ci.yml`, `audit.yml`) is green on that commit.
## Distribution channels & required secrets

A tagged release drives **eight** channels. They are all wired and locally
verified; each registry push is gated on an operator secret, and the build for
that channel still runs (producing artifacts) even when the secret is absent â€”
only the push is skipped. Set the secret to light the channel up.

| Channel | Built by | Push gated on |
| --- | --- | --- |
| Prebuilt binaries, `curl \| sh`, PowerShell | cargo-dist (`release.yml`) | nothing â€” attaches to the GitHub Release |
| Docker / GHCR | `docker.yml` | nothing â€” built-in `GITHUB_TOKEN` |
| Homebrew | cargo-dist `publish-homebrew-formula` | a tap repo (`dshakes/homebrew-firstpass`) **+ `HOMEBREW_TAP_TOKEN`** |
| npm | cargo-dist `publish-npm` | an owned `@firstpass` scope **+ `NPM_TOKEN`** |
| pip / uvx (wheels) | maturin (`python-wheels.yml`) | **`PYPI_API_TOKEN`** |
| crates.io (`cargo install firstpass-proxy`) | `cargo publish` (manual) | a crates.io token; publish `firstpass-core` **before** `firstpass-proxy` (path-dep order) |

`cargo install --git` and Docker `:latest` (on `main`) already work with no
release. The `dist plan` / build steps below cover the cargo-dist channels; the
maturin wheels build in parallel off the same `v*` tag.

## 1. Confirm the plan

Run `dist plan` from the repo root and read the output â€” it tells you exactly
what `dist` thinks it will build for the next release without touching
anything:

```bash
dist plan
```

**How you know it worked:** the output lists the workspace version
(`Cargo.toml` `[workspace.package] version`, currently `0.1.0`), the release
target triples from `dist-workspace.toml`
(`aarch64-apple-darwin`, `aarch64-unknown-linux-gnu`, `x86_64-apple-darwin`,
`x86_64-unknown-linux-gnu`, `x86_64-pc-windows-msvc`), and the app(s) it will
package. If `dist plan` errors, fix that before touching a version or a tag â€”
the CI run will hit the same error.

## 2. Bump the version

Bump `version` in `[workspace.package]` in the root [`Cargo.toml`](../../Cargo.toml)
(every crate under `crates/*` inherits it via `version.workspace = true`).
Commit that as a normal PR-reviewed change, merged to `main` like any other:

```bash
git checkout -b release/vX.Y.Z
# edit Cargo.toml: version = "X.Y.Z"
cargo fmt --all --check   # confirm the bump didn't touch formatting
git commit -am "chore: bump version to X.Y.Z"
git push -u origin release/vX.Y.Z
# open a PR, get it reviewed and merged to main
```

## 3. Re-run the plan against the merged commit

```bash
git checkout main && git pull
dist plan
```

**How you know it worked:** `dist plan` now reports the new `X.Y.Z` version.

## 4. Tag and push â€” the human-gated trigger

`release.yml` triggers on a version-shaped tag push (see the `on:` block in
that file). Tag the exact commit that merged the version bump:

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

This is the irreversible, outward-facing step: pushing the tag starts a
public GitHub Actions run that, on success, publishes a GitHub Release with
downloadable artifacts. Don't do it as a side effect of anything else, and
don't script it into an unattended pipeline.

## 5. Watch the release workflow

Open the Actions tab (or `gh run watch`) for the `release` workflow run
triggered by the tag push. It builds each target, generates checksums and
installer scripts, and (only after the plan/build/host jobs succeed)
publishes the GitHub Release.

**How you know it worked:** the workflow run is green, and the release page
for `vX.Y.Z` lists an artifact per target triple plus `shell`/`powershell`
installer scripts and a checksums file.

## 6. Verify artifacts and checksums

Download an artifact for your platform and verify it against the published
checksum before telling anyone it's ready:

```bash
curl -LO https://github.com/<org>/firstpass/releases/download/vX.Y.Z/<artifact-name>
curl -LO https://github.com/<org>/firstpass/releases/download/vX.Y.Z/<artifact-name>.sha256
shasum -a 256 -c <artifact-name>.sha256
```

**How you know it worked:** `shasum` prints `OK`. Then actually run the
binary â€” `firstpass doctor` against a known-good config â€” before considering
the release verified, not just downloaded.

## 7. Registry channels (Homebrew, npm, pip, crates.io)

`homebrew` and `npm` are already in `installers` (`dist-workspace.toml`) and
their publish jobs are in `release.yml`; pip/uvx wheels build via
`python-wheels.yml`. Each is a no-op until its secret exists (see *Distribution
channels & required secrets* above). To light one up:

- **Homebrew:** create the tap repo `dshakes/homebrew-firstpass`, add a
  `HOMEBREW_TAP_TOKEN` secret with push access to it. Next tag publishes the
  formula; users then `brew install dshakes/firstpass/firstpass`.
- **npm:** own the `@firstpass` scope on npmjs, add an `NPM_TOKEN` secret.
- **pip / uvx:** add a `PYPI_API_TOKEN` secret; the tag build publishes wheels
  for all five target platforms, then `pip install firstpass` / `uvx firstpass`.
- **crates.io:** manual and ordered â€” `cargo publish -p firstpass-core` first
  (a `firstpass-core` version must exist on crates.io before `firstpass-proxy`
  resolves), then `cargo publish -p firstpass-proxy`. `firstpass-bench` is
  `publish = false`.

## Rollback: pin to a prior version

There is no "undo" for a published GitHub Release short of manually marking
it a draft/pre-release or deleting it (deleting a public release is a
destructive, outward-facing action â€” confirm with whoever owns the repo
before doing it, and prefer *not* deleting: yank forward with a fixed
release instead of erasing history people may already be pinned to).

To roll a deployment back to a known-good prior version:

1. Identify the last good tag (e.g. `vX.Y.Z-1`).
2. Re-download that release's artifact directly from
   `https://github.com/<org>/firstpass/releases/tag/vX.Y.Z-1` and verify its
   checksum the same way as step 6.
3. Redeploy that binary. If the bad release already reached production,
   also flip to observe mode while you sort it out â€” see
   [`docs/runbooks/soak.md`](soak.md) rollback section.
4. If the broken release must not be installed by new users going forward,
   mark it `Pre-release` (not delete) in the GitHub UI and note the issue in
   the release notes of the next good tag.
