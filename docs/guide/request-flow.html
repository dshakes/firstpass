Request flow

How a prompt in Claude Code — or a call from the Claude Agent SDK — actually reaches the router. One idea carries the whole page: Firstpass sits below the agent, so it never sees a conversation, a turn, or an agent. It sees a stream of independent HTTP requests. Simple questions, complex reasoning, subagents, agent teams, and dynamic workflows are all just more of the same request. This page follows one from the terminal to the served answer, then builds up to the hard cases.

Read this after "How it works" That page explains the gate and the ladder — the machinery. This page explains what a request is and how your prompts turn into requests, so the routing you read about elsewhere has something concrete to route.

The one idea#

Firstpass is an HTTP proxy. Your agent — Claude Code, the Agent SDK, or anything that speaks the Anthropic or OpenAI wire format — points its base URL at Firstpass instead of the provider. From that seat, Firstpass sees exactly one kind of thing: a completion request, a single POST with a body of messages and tools. It does not see "your session," "turn 4," or "the planner subagent." Those are concepts that live above it, inside the agent runtime.

Everything in this page is a consequence of that seat. The agent loop, subagents, and multi-agent workflows are all built by the client out of many independent requests — and Firstpass routes each one on its own merits.

The load-bearing sentence A terminal prompt is not a request. It is the start of an agent loop that emits many requests. Firstpass routes requests, not prompts.

The unit of routing is one request#

When you type a prompt and hit enter, the agent begins a loop: think, call a tool, read the result, think again, maybe spawn a subagent, and eventually answer. Each "think" is a separate model call — a separate POST to Firstpass. A single prompt routinely produces five, ten, or fifty requests. Each arrives independently, is routed independently, and is proven independently.

What a request looks like#

A request is a standard provider call. Firstpass accepts two wire dialects and matches whichever the client speaks:

FieldWhat it is
POST /v1/messagesAnthropic dialect — what Claude Code and the Agent SDK send.
POST /v1/chat/completionsOpenAI dialect — for OpenAI-shaped clients.
"model" in the bodyA real model, or the sentinel "auto". On the enforce path this is overwritten by the rung the router picks — see the lifecycle.
messages / toolsThe full prompt and the entire conversation history so far. The client sends it every turn — the model is stateless, so history rides in the body.

Five optional headers are the only channel through which the client tells Firstpass anything beyond the raw call. They are how routing gets its signal:

HeaderPurpose
x-firstpass-agentWhich agent this call belongs to. Becomes a routing feature.
x-firstpass-subagentWhich subagent (e.g. architect, test-runner). Becomes a routing feature.
x-firstpass-modePer-request mode override (cost, quality, …). Highest-precedence mode signal.
x-firstpass-sessionGroups requests into one session — for the audit trail and the per-session budget cap. The only conversation-aware knob.
x-firstpass-keyTenant key, when multi-tenant auth is on.
If you send no headers Firstpass still works — every call falls to the wildcard route (match = {}). The headers don't enable routing; they let you route different traffic to different ladders. No tags means one ladder for everything.

The life of one request#

Every POST — no matter what produced it — travels the same four steps. There is no fast path for "simple" prompts and no special case for subagents; the uniformity is the point.

  1. Extract features. From the headers (agent, subagent) plus cheap body-derived signals — tool count, whether images are present, a coarse request-size bucket, an hour bucket. Firstpass never reads your prompt text to route; it buckets by byte size, which keeps the routing signal privacy-preserving.
  2. Match a route. route_for(features) walks your [[route]] blocks and returns the first whose match predicate holds. Every field you set in match is an AND-constraint; every field you omit is a wildcard; match = {} catches everything.
  3. Enforce or observe. The matched route runs in one of two families of mode. Enforce runs the cascade: open on the cheapest rung, run the gate, serve if it passes, else climb one rung — overwriting the request's model with the rung's model as it goes. Observe forwards the original request unchanged (also the safe fallback when a request can't be routed faithfully).
  4. Serve and record. The winning output is returned in the caller's own dialect. The full decision — every rung tried, every gate verdict, the cost — is written to a hash-chained receipt.
Where the "which model answered" answer lives The response you get back is a normal completion in your dialect — Firstpass does not bolt a "served by rung 2" header onto it. The rung, the gate verdicts, and the cost live in the receipt, the auditable record. That is deliberate: the answer is the product; the proof is the receipt.

A simple question#

Start with the easy case. You ask a one-shot question that needs no tools. The agent sends one request; Firstpass opens it on rung 0 (the cheapest model), the gate passes, and that output is served. One request, one rung, done.

bash
# the model can be "auto" — the matched route's ladder decides anyway
curl localhost:8080/v1/messages \
  -d '{"model":"auto","messages":[{"role":"user","content":"Capital of France?"}]}'

The receipt for that request has a single attempt on rung 0 with served: true. No escalation, no waste. The whole cascade for an easy request is the cheapest model — the gate just confirmed it was good enough to ship.

A complex question#

Now a real task: "find the bug in this handler and fix it." The agent can't answer in one shot — it reads files, reasons, edits, re-reads, and concludes. That is a loop, and every iteration is its own request:

  • Request 1 — reason about the task, decide to read handler.rs.
  • Request 2 — with the file contents now in the messages, reason about the bug.
  • Request 3 — propose an edit.
  • Request 4 — with the edit result in the messages, confirm the fix.

Firstpass routes each one independently and proves each one independently. Request 2 might escalate to a stronger rung because reasoning over a big file failed the gate; request 4 might pass on rung 0 because "did the edit apply cleanly" is easy. The router never assumes that because turn 2 was hard, turn 4 will be — it re-earns the answer every time. That is the whole reason the cascade is safe to run per-request: a stale routing decision can never be carried forward, because none is kept.

Subagents#

When the main agent spawns a subagent, the subagent runs its own loop — and every call in that loop is, again, an independent POST to Firstpass. Nothing about a subagent call is structurally different from a main-loop call. So how does Firstpass route a planner subagent to a strong model and a mechanical one to a cheap model?

You tag the request. The orchestrator sets x-firstpass-subagent on the subagent's calls, and a route matches on it:

toml
# the planner: give it a tall ladder and a quality bias
[[route]]
match = { subagent = "architect" }
mode  = "quality"
ladder = ["anthropic/claude-haiku-4-5", "anthropic/claude-sonnet-5", "anthropic/claude-opus-4-8"]

# a mechanical subagent: one cheap rung is plenty
[[route]]
match = { subagent = "test-runner" }
mode  = "cost"
ladder = ["anthropic/claude-haiku-4-5"]

# everything else — the wildcard catch-all, always last
[[route]]
match = {}
ladder = ["anthropic/claude-haiku-4-5", "anthropic/claude-sonnet-5"]
The honest boundary Firstpass can only tell a subagent call apart from a main-loop call if the client sets the header. They are otherwise byte-identical POSTs. Out of the box, a client that doesn't emit x-firstpass-subagent sends everything to the wildcard route — which is correct and safe, just not differentiated. Subagent-aware routing is a capability you switch on by tagging, not something Firstpass can infer from the wire.

Agent teams & dynamic workflows#

A team of agents — or a dynamic workflow that fans out dozens of workers in parallel — looks, from Firstpass's seat, like nothing new: more concurrent requests. Ten agents running at once is ten streams of independent POSTs. Each is matched on its own features (agent, subagent, plus task_kind and language if your client sets them) and routed to the ladder you configured for that kind of worker. There is no "team" object and no coordinator inside the router; concurrency is just volume.

What ties a team together is not routing — it's budget. Give every request in a workflow the same x-firstpass-session, and the per-session cap (per_session_usd) governs the whole team's spend as one pool. That is the one place a multi-agent workload is treated as a unit, and it is a spend guardrail, not a routing input.

Dynamic ≠ special A workflow that decides at runtime to spawn a new worker doesn't need Firstpass to know that in advance. The new worker's first call is just another request; it matches a route the instant it arrives. Dynamism lives in the orchestrator; Firstpass only ever sees the next POST.

Multi-turn & "previous context"#

This is the question that trips people up, so here is the blunt version: the router does not remember your conversation, and that is on purpose.

Where does the context go, then? Into the request body. An LLM is stateless — every turn, the client (Claude Code, the SDK) resends the entire message history so the model can "remember." That history is part of the POST Firstpass forwards. So the context absolutely reaches the model — it just rides in the prompt, not in any router-side memory.

Firstpass itself keeps no per-conversation routing state. Turn 12 re-enters the cascade at the same start rung as turn 1 and is proven from scratch. Two things, and only two, thread across the calls of one conversation:

ThreadWhat it carries across turns
x-firstpass-session → budgetA spend cap and an audit grouping. This is the only conversation-aware knob. It never changes which model is picked — only whether the session has money left.
Learned start-rung (optional)An off-by-default bandit (ADR 0007) can learn a smarter entry rung for a class of traffic — keyed by request features, not by conversation. It moves where the climb begins; the gate still decides what ships, so it can never change correctness.
The real cost ceiling Because history rides in every request, a long conversation makes every rung's input larger — including the cheap first pass and each escalation. The cascade does not summarize or dedupe your history; that is the client's job. On very long sessions this input-token inflation, not the routing, is what dominates cost. Name it, measure it, and cap it with per_session_usd.

Claude Code vs the Agent SDK#

Both are just HTTP clients that speak the Anthropic dialect, so both wire up the same way: point the base URL at Firstpass. Everything on this page applies identically to either — the differences are only in how you set the base URL and headers.

bash
# Claude Code — send every call through Firstpass instead of the API
export ANTHROPIC_BASE_URL=http://localhost:8080
# now `claude` runs its whole loop — reasoning, tools, subagents — through the router
ts
// Claude Agent SDK — same idea, set the baseURL on the client
const client = new Anthropic({ baseURL: "http://localhost:8080" });
// optional: tag calls so subagent-aware routes can fire
// headers: { "x-firstpass-subagent": "architect" }

Two honest notes on wiring:

  • Enforce vs observe is your call. Start in observe — Firstpass forwards every request untouched and records what it would have done, so you get the receipts and the savings estimate with zero behavior change. Flip to an enforce mode when the shadow data convinces you.
  • Tagging is opt-in. Neither Claude Code nor the SDK emits x-firstpass-agent/-subagent on its own. Until you inject those headers (a wrapper, a proxy shim, or SDK-level request options), all traffic matches the wildcard route. That is a fine place to start — differentiate later.

How Firstpass tackles all of it#

Every scenario on this page reduces to the same handful of moves. The power is in how little there is:

You have…Firstpass sees…You steer it with…
A simple questionOne request → rung 0 + gateNothing — the default just works
A complex taskMany independent requests, each re-provenThe ladder + gate per route
SubagentsTagged requests matched to their own routex-firstpass-subagent + [[route]]
Agent teams / workflowsMore concurrent requests, matched individuallyPer-worker routes + shared session budget
Multi-turn conversationIndependent turns; history rides in the bodyx-firstpass-session → per_session_usd

The design holds because it refuses to be clever about state. It does not predict what your conversation will need; it proves each request against a gate and serves the cheapest output that clears it. Statelessness is what makes that safe to run on every call, differentiation comes from the tags you choose to send, and the one thing carried across a conversation — the budget — is a guardrail, never a guess. That is the whole model: proof over prediction, one request at a time.
