how it works

Four moves. One receipt.

▸

Route

Open on the cheapest rung of a declarative ladder — across any provider, incl. open-source (Anthropic, OpenAI, Groq, DeepSeek, local Ollama / vLLM …). Drop-in: your agent already speaks the wire format, so just point base_url at it. BYOK, keys redacted from every log.

◈

Prove

Run a real gate on the real output — your tests, a JSON schema, a fresh-context judge (maker ≠ checker), or k-sample self-consistency. Meet you where you are: start with none, add one line when ready.

↑

Escalate

One rung up on failure, budget-capped; a provider 5xx fails over — to a different provider if the ladder says so. The decision seals into a tamper-evident hash-chain.

↻

Learn

Outcomes feed back. The serve threshold self-tunes from live results so the guarantee tracks your traffic as it drifts — no retraining, ever.

the receipt

Not a dashboard number.
A decision you can audit.

Every call becomes a hash-chained JSON trace. Point at any request and answer why did this go to that model, and what did it cost — then hand the log to an auditor who re-derives the whole chain with no proxy and no database.

SEALED ✓
{
  "trace_id": "0192f3a1-7c4e-7abc", "prev_hash": "9f2c…a1b7",   // chains to the prior decision — tamper-evident
  "attempts": [
    { "rung":0, "model":"…/haiku-4-5",  "gate":"fail", "cost":0.0007 },  // cheap first — gate caught it
    { "rung":1, "model":"…/sonnet-5",  "gate":"pass", "cost":0.0121 }   // escalated, proven, served
  ],
  "final": { "served_rung":1, "savings_usd":0.0502 }   // vs always-top-tier

firstpass export + verify

Export the sealed log as JSONL; anyone re-derives the hash chain from genesis on their own machine. A single altered or reordered receipt breaks the chain at its index and exits non-zero. The EU-AI-Act-style logging story, built in.

Durable, never-drop

FIRSTPASS_RECEIPTS=durable spills receipts to disk under backpressure and drains them on boot with the chain verified valid — no decision is ever silently lost.

on the near horizon

One dial: how hard should
this one try?

◇ designed · validating

Cost, balanced, quality, latency, or max — a mode is a constraint the router honors, chosen per task, per turn, even per request with one header. Same guarantee underneath; the mode just moves where on the cost/quality frontier you sit.

Cost. Open on the cheapest rung and stay there as long as the gate clears. Escalate only on hard proof of need. The default posture — most requests never leave the cheap tier.

Balanced. Let the learned start-rung open a step higher when your own receipts say the cheap model usually fails this kind of task — trading a little spend for fewer escalations.

Quality. Raise the serve threshold and prefer stronger rungs. Fewer marginal passes served; the guarantee target tightens.

Latency. Fire speculative escalation in the marginal zone so the next rung is already in flight if the gate fails — pay parallel tokens to buy tail latency.

Max. Open at the top of the ladder. For the rare request where the answer has to be right the first time and cost is no object.

# choose the posture per request — one header
curl $BASE/v1/messages \
  -H "x-firstpass-mode: cost" \
  -d @req.json

Honest status: modes are designed, not yet shipped. They land behind a validation gate — a mode is only worth exposing if it moves the cost/quality frontier measurably on your traffic, and that's the study that promotes each one from research to default.

the research

Verify proportional to doubt.

◇ research · go/no-go gated

Today Firstpass proves before it serves. The next question is how much proof each answer needs. An easy request that three signals already agree on shouldn't pay for a full test run; a genuinely uncertain one should. That's elastic verification: spend verification budget as a function of measured doubt — with a conformal guarantee over the verify-or-skip decision itself, so cutting a check is never a silent risk.

⌁

Value-of-information gating

Estimate what a verification step is worth before running it — skip the checks that can't change the decision, spend on the ones that can.

◎

Per-query difficulty

Predict how hard a request is likely to be for the cheap rung, from the prompt and cheap signals — to choose where to start and how much to check.

⟿

Session-level routing

Route over a whole trajectory, not one turn — carry difficulty and outcome context across an agent's session.

How we'll know it's real: each of these ships only if a pre-registered go/no-go study shows it beats verify-everything on cost at an equal-or-better served-failure bound. Same rule that promoted the start-rung bandit from idea to code. Until then it's on the roadmap, honestly labeled — not in the product.

One command onboards you

firstpass onboard --apply detects your shell and agents, starts the proxy in observe mode (zero config, zero risk), routes your agent, and verifies itself. Idempotent, dry-run by default; firstpass offboard undoes all of it.
